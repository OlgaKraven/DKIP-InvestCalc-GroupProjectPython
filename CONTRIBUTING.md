## Руководство по участию в проекте InvestCalc

Спасибо за интерес к проекту **InvestCalc — Инвестиционный аналитик ИС**.
Этот репозиторий используется в учебных целях, но мы стремимся поддерживать
в нём практики, близкие к профессиональной разработке.

---

## 1. Как можно помочь

Вы можете:

- сообщать об ошибках через **Issues**;
- предлагать улучшения и новые функции;
- дополнять документацию (в т.ч. учебные материалы);
- улучшать покрытие тестами;
- дорабатывать DevOps-часть (CI/CD, Docker, мониторинг).

---

## 2. Рабочий процесс (GitHub Flow)

1. Сделайте **fork** репозитория и склонируйте его локально.
2. Создайте отдельную ветку от `main`:

   ```bash
   git checkout -b feature/my-feature
   ```

   Рекомендуемый формат имени ветки:

   - `feature/...` — новая функциональность;
   - `fix/...` — исправление ошибок;
   - `docs/...` — изменения в документации;
   - `devops/...` — DevOps и инфраструктура.

3. Внесите изменения и добавьте тесты (если применимо).
4. Убедитесь, что локально проходят проверки:

   ```bash
   pytest
   mkdocs build
   ```

5. Сделайте коммиты с понятными сообщениями:

   ```bash
   git commit -m "feat: add ROI sensitivity report"
   ```

   Рекомендуется придерживаться стиля **conventional commits** (`feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`).

6. Отправьте ветку в свой форк и создайте **Pull Request** в ветку `main` исходного репозитория.
7. Заполните шаблон pull request: опишите, что было сделано, как тестировалось и какие есть ограничения.

---

## 3. Код-стайл и качество

- Код пишется на **Python 3.11+**.
- Для форматирования рекомендуется использовать **black** и **isort**.
- Для статического анализа — **ruff** или **flake8** (по возможности).
- Желательно придерживаться принципов чистой архитектуры:
  разделение доменной логики, сервисов и API.
- Все новые функции должны сопровождаться:
  - тестами (модульными и/или интеграционными),
  - минимальным описанием в документации (`docs/`).

---

## 4. Документация

- Основной текст документации — в каталоге `docs/`, структура уже задана.
- Перед добавлением нового раздела подумайте, к какому блоку он относится:
  требования, архитектура, API, DevOps, качество и т.д.
- Диаграммы рекомендуется оформлять в **Mermaid** или PlantUML (если поддерживается конфигурацией).

После изменения документации убедитесь, что локально собирается сайт:

```bash
mkdocs build
```

---

## 5. Issues и обсуждения

Перед созданием нового issue:

- проверьте, нет ли уже похожей проблемы или предложения;
- чётко опишите контекст (какой сценарий, какие входные данные, что ожидалось, что получилось);
- при необходимости приложите скриншоты, логи, ссылку на ветку или commit.

Для багов и фич используйте соответствующие шаблоны в разделе **Issues**.

---

## 6. Pull Requests

При создании PR:

- давайте **понятное название**, отражающее суть изменений;
- кратко опишите изменения в основном блоке;
- укажите, на что может повлиять этот PR (API, модель данных, документация, Docker и т.п.);
- добавьте ссылку на связанные issues (например, `Closes ##42`);
- убедитесь, что проходят проверки CI (GitHub Actions).

---

## 7. Для преподавателей и студентов

Если репозиторий используется в рамках учебного курса:

- не публикуйте в открытый доступ персональные данные и закрытые материалы;
- оформляйте учебные ветки/PR так, чтобы по ним было легко проверять прогресс;
- по возможности оставляйте комментарии в Issues / PR для обучения и обратной связи.

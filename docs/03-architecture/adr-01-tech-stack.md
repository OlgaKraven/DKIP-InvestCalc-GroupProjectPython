# ADR-01 — Выбор технологического стека для системы InvestCalc

Дата: 2025-01-01  
Статус: Принято

---

## 1. Контекст

Проект InvestCalc — учебная информационная система, предназначенная для расчёта экономической эффективности ИС.  
Система должна:

- быть простой для студентов СПО;
- позволять быстро разрабатывать REST API;
- иметь встроенную валидацию данных;
- легко развертываться (локально + Docker);
- поддерживать автодокументацию API;
- иметь минимальные требования к инфраструктуре.

Необходимо выбрать технологический стек, который будет использоваться для разработки всего проекта.

---

## 2. Решение

В качестве стека разработки выбраны:

- **Язык:** Python 3.12  
- **Фреймворк:** FastAPI  
- **Сервер:** Uvicorn (ASGI)  
- **Валидация моделей:** Pydantic  
- **Документация API:** Swagger UI + Redoc (автоматически генерируется FastAPI)  
- **Тестирование:** pytest  
- **Контейнеризация:** Docker  
- **Сборка и CI/CD:** GitHub Actions  
- **Документация:** MkDocs  

---

## 3. Обоснование выбора

## 3.1. FastAPI
- современный, быстрый и лёгкий фреймворк;
- отлично подходит для учебных проектов;
- минимальный код для создания API;
- из коробки предоставляет Swagger и Redoc;
- синхронная и асинхронная работа;
- простая валидация через Pydantic.

FastAPI — идеальный баланс между простотой и современностью.

### 3.2. Python
- распространён среди студентов СПО;
- прост в освоении;
- богатая экосистема библиотек;
- не требует сложной инфраструктуры.

### 3.3. Pydantic
- строгая типизация входных/выходных данных;
- автоматическая валидация;
- минимальная вероятность ошибок.

### 3.4. Docker
- гарантированная повторяемость среды;
- возможность работать на Windows/Linux/Mac;
- легко интегрируется в CI/CD.

### 3.5. GitHub Actions
- бесплатное CI для open-source/учебных репозиториев;
- простая конфигурация YAML.

### 3.6. MkDocs
- быстрый генератор документации;
- отлично подходит для больших учебных проектов;
- поддерживает Mermaid.

---

## 4. Альтернативы

### 4.1. Django REST Framework
**Почему отклонён:**
- слишком тяжёлый для учебной задачи;
- требует ORM и структуры проекта «монолит».

### 4.2. Flask
**Почему отклонён:**
- нет встроенной валидации;
- нет автоматической API-документации;
- больше ручной работы.

### 4.3. Node.js + Express
**Почему отклонён:**
- высокая сложность для студентов;
- нет встроенной строгой типизации.

### 4.4. C## ASP.NET Core
**Почему отклонён в этом проекте:**
- используется в других учебных проектах;
- требует тяжёлой инфраструктуры для быстрого старта.

---

## 5. Последствия

### Положительные:
- быстрый старт разработки;
- простота обучения студентов;
- минимальное количество boilerplate-кода;
- высокая скорость API;
- автоматическая документация;
- лёгкость контейнеризации.

### Негативные:
- Python медленнее Go/Java (не критично для учебной ИС);
- Pydantic может потребовать дополнительных знаний типизации;
- для PDF-генерации потребуется использование сторонних библиотек.

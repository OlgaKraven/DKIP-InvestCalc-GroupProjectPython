# Проектный Устав (Project Charter)

Проект: **InvestCalc — Инвестиционный аналитик ИС**  
Тип проекта: учебный, командный, проектирование и разработка мини-ИС  
Версия: 1.0  
Дата: <указать дату>

---

## 1. Назначение документа
Проектный устав определяет цели, границы, ключевые требования, роли и ресурсы проекта **InvestCalc**.  
Документ служит формальной точкой запуска проекта, обеспечивает общее понимание всеми участниками и используется для согласования объёма работ.

---

## 2. Обзор проекта

### 2.1. Описание проекта  
**InvestCalc** — это учебная мини-информационная система для оценки экономической эффективности ИТ-проектов.  
Система позволяет рассчитывать:

- TCO (Total Cost of Ownership),
- ROI (Return on Investment),
- Payback Period (срок окупаемости),
- анализ чувствительности ±N%,
- работу со сценариями через JSON-файлы.

Проект разрабатывается как FastAPI/ASP.NET сервис с документацией, тестами и DevOps-практиками.

---

## 3. Бизнес-обоснование (Business Case)
Современные компании требуют объективной оценки вложений перед внедрением информационных систем.  
Учебный проект позволяет студентам:

- освоить технику расчёта эффективности ИТ-проектов,
- получить опыт проектирования API и бизнес-логики,
- практиковать DevOps (Docker, healthchecks),
- применять стандарты документации (ТЗ, Use Case, User Story, Тест-план).

**В учебной среде** проект формирует цифровое портфолио и имитирует реальный процесс разработки.

---

## 4. Цели проекта

### 4.1. Основные цели
1. Разработать рабочий API-сервис InvestCalc с расчётами TCO/ROI/Payback.  
2. Реализовать анализ чувствительности ±20% и хранение сценариев в JSON.  
3. Создать полный комплект проектной документации.  
4. Настроить Docker-контейнеризацию и базовые DevOps-практики.  
5. Подготовить итоговую отчётность и провести демонстрацию проекта.

### 4.2. Измеримые результаты (SMART)
- **API** работает, документы Swagger доступны → `/docs`.
- В сервисе корректно проходят **расчёты**.
- Все эндпоинты API покрыты **автотестами**.
- Проект поднимается через Docker не более чем за **30 секунд**.
- Полный комплект документации сформирован в `docs/`.

---

## 5. Границы проекта (Scope)

### 5.1. Входит в проект
- backend-сервис (FastAPI/ASP.NET Core),
- расчёт экономических показателей,
- анализ чувствительности,
- JSON-хранилище сценариев,
- Swagger / OpenAPI,
- тесты (unit, integration),
- Dockerfile, healthchecks,
- проектная документация (ТЗ, Use Cases, User Stories, C4-модель, тест-план и др.),
- отчётность (практика, групповой проект).

### 5.2. Не входит в проект
- полноценный UI веб-клиента (может быть опционально),
- внешняя база данных (используем JSON),
- сложные финансовые алгоритмы,
- продвинутая авторизация/аутентификация,
- полноценный DevOps (CI/CD включён частично).

---

## 6. Основные требования (High-Level Requirements)

1. Система должна принимать входные данные и возвращать корректные финансовые показатели.
2. Должны быть реализованы эндпоинты:
   - `POST /api/v1/calc`
   - `POST /api/v1/sensitivity`
   - `GET /api/v1/scenarios`
   - `GET /api/v1/scenarios/{id}`
   - `POST /api/v1/scenarios`
3. Логика расчётов должна быть вынесена в отдельный слой (`services/`).
4. API должен быть документирован автоматически (Swagger).
5. Проект должен подниматься локально и в Docker.
6. В системе должны быть автотесты:
   - тесты расчётов,
   - тесты чувствительности,
   - интеграционные тесты API.

---

## 7. Ограничения

- Время выполнения проекта — ограничено учебным расписанием (3–4 недели).
- Команда 2–4 человека.
- Хранилище данных — JSON, без SQL/NoSQL.
- Минимальный UI (или полное его отсутствие).
- Требования к качеству кода — учебные, но с применением реальных практик.

---

## 8. Предположения

- Каждый участник владеет базовыми навыками Python/C## и Git.
- Все участники могут работать удалённо через GitHub.
- Аппаратные ограничения минимальны — достаточно ПК и Docker Desktop.
- Студенты готовы работать спринтами и вести отчётность.

---

## 9. Критерии успешности проекта (Definition of Done)

1. API доступен и успешно выполняет основные расчёты.  
2. Проект собирается через Docker без ошибок.  
3. Все автотесты проходят.  
4. Документация полная и структурированная.  
5. Защита проекта проведена, вопросы экспертов закрыты.  
6. Репозиторий оформлен по стандартам (README, templates, docs).

## 12. Риски (см. подробный реестр в `risk-register.md`)
Критические риски:
- отсутствие автотестов → неверные расчёты,
- несогласованность команды → дублирование работ,
- проблемы с Docker → невозможность демонстрации,
- потеря/повреждение JSON → невозможность восстановления сценариев.

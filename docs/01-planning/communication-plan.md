Версия: 1.0
Дата: <указать>

---

## 1. Назначение документа

План коммуникаций определяет:

* кого,
* о чём,
* в каком формате,
* как часто

нужно информировать в рамках проекта **InvestCalc**.

Документ обеспечивает прозрачность, уменьшает риски, помогает команде работать согласованно и эффективно.

---

## 2. Цели коммуникационного плана

1. Установить единые правила взаимодействия команды.
2. Повысить эффективность разработки за счёт своевременного обмена информацией.
3. Обеспечить методиста и экспертов полной и корректной информацией о ходе проекта.
4. Стандартизировать форматы отчётности, документов и консультаций.

---

## 5. Форматы коммуникаций

### 5.1. Стенданы (daily / short sync)

Проводятся в чате или короткой встрече (5–10 минут).
Каждый участник сообщает:

1. Что сделал вчера
2. Что делает сегодня
3. Есть ли блокеры

### 5.2. Технические обсуждения

* GitHub Issues → постановка задач
* Pull Requests → обсуждение кода
* Project Board → визуальное состояние задач

### 5.3. Документирование

Все артефакты фиксируются в `/docs/`:

* ТЗ
* Use Case
* User Stories
* C4
* Тест-план
* Отчёты

UX — в Figma, ссылка размещается в `docs/`.

### 5.4. Коммуникации с методистом

* Передаются промежуточные документы (ТЗ, архитектура, тест-план).
* Проводятся консультации 1 раз в неделю.

### 5.5. Коммуникации с экспертами

* По готовому проекту.
* Формат: демонстрация, ответы на вопросы.

## 6. Еженедельный цикл коммуникаций (Sprint Rhythm)

### Понедельник

* Аналитика, уточнение задач
* Обновление Roadmap

### Вторник — Среда

* Активная разработка
* PR, ревью, тесты

### Четверг

* Промежуточная сборка
* Проверка Docker
* Обновление документации

### Пятница

* Мини-демо (внутри команды)
* Отправка методисту

### Суббота / Воскресенье (при необходимости)

* Доработки
* Подготовка к следующему спринту

---

## 7. Правила коммуникаций в команде

1. **Все задачи создаются через GitHub Issues.**
2. **Все изменения проходят через Pull Requests.**
3. **Нельзя пушить прямо в main.**
4. **Каждый PR сопровождается кратким описанием и/или ссылкой на Issue.**
5. **Документация не хранится в чатах — только в репозитории.**
6. **Споры решаются на встрече, не в длинных переписках.**
7. **Если задача больше 4 часов — её нужно декомпозировать.**

---

## 8. Эскалация вопросов

* **Технические проблемы** → Backend → TL
* **QA/DevOps проблемы** → QA/DevOps → TL
* **UX проблемы** → UX/UI → TL
* **Организационные вопросы** → TL → методист
* **Критические проблемы** → методист → администрация (если требуется)

---

## 9. Хранение и актуализация документа

* Документ хранится в `docs/01-planning/communication-plan.md`.
* Обновляется Team Lead или участником, ответственным за организацию.
* Изменения отслеживаются через Git.

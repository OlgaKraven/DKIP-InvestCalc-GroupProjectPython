# Карта стейкхолдеров

Проект: **InvestCalc — Инвестиционный аналитик ИС**
Тип проекта: учебный, командный
Версия: 1.0
Дата: <указать>

---

## 1. Назначение документа

Карта стейкхолдеров определяет всех участников проекта, их интересы, влияние на проект и уровень вовлечённости.

Документ используется для:

* планирования коммуникаций,
* распределения ответственности,
* анализа рисков и ожиданий.

---

## 2. Стейкхолдеры проекта

Стейкхолдеры классифицированы по уровням: **внутренние**, **внешние**, **вспомогательные**.

---

### 2.1. Внутренние стейкхолдеры (команда проекта)

| Роль                    | Описание                                                     | Интересы                                                 | Влияние        | Вовлечённость |
| ----------------------- | ------------------------------------------------------------ | -------------------------------------------------------- | -------------- | ------------- |
| **Team Lead**           | Руководит командой, контролирует архитектуру, проводит ревью | Успешная защита, качество решения, порядок в репозитории | Высокое        | Высокая       |
| **Backend-разработчик** | Реализует API, бизнес-логику расчётов                        | Чистый код, понятная архитектура, адекватные требования  | Среднее        | Высокая       |
| **QA/DevOps**           | Тест-план, авто-тесты, CI/CD, Docker                         | Качество, стабильность, воспроизводимость                | Среднее        | Высокая       |
| **UX/UI Designer**      | Прототип интерфейса, проработка пользовательского опыта      | Понятность UI, визуальная целостность                    | Низкое–среднее | Средняя       |

---

### 2.2. Внешние стейкхолдеры

| Роль                                              | Описание                                  | Интересы                                               | Влияние | Вовлечённость |

| **Методист / Преподаватель**                      | Проверяет и оценивает проект              | Документация, корректность расчётов, учебная ценность  | Высокое | Средняя       |

## 5. Стратегия взаимодействия со стейкхолдерами

| Стейкхолдер             | Стратегия                 | Формат взаимодействия     | Частота               |
| ----------------------- | ------------------------- | ------------------------- | --------------------- |
| **Team Lead**           | Управлять, координировать | стендапы, ревью           | 2–3 раза в неделю     |
| **Backend-разработчик** | Сотрудничество            | Git, код-ревью, задачи    | ежедневно             |
| **QA/DevOps**           | Вовлечение                | тест-план, CI, Docker     | ежедневно             |
| **UX/UI**               | Информирование            | обсуждение прототипов     | по мере необходимости |
| **Методист**            | Управлять ожиданиями      | сдача документации        | 1 раз в неделю        |
| **Эксперт**             | Информировать             | презентация перед защитой | 1 раз за проект       |
| **Персона-заказчик**    | Информировать             | уточнение требований      | 1–2 встречи           |
| **Учебная группа**      | Мониторинг                | демонстрация              | необязательно         |

## 6. План коммуникаций

* **Команда** — общий чат, GitHub Issues/PR, регулярные стендапы.
* **Методист** — передача промежуточных версий документации.
* **Эксперты** — подготовка демо и презентации к финалу.
* **UX/UI** — обмен макетами через Figma, обсуждение в чате.
* **Персона-заказчик** — работа при уточнении требований и сценариев.

Рекомендуется создать: `communication-plan.md`.

---

## 7. Ключевые ожидания стейкхолдеров

* **Методист**: полная документация, учебная ценность.
* **Эксперты**: корректные расчёты, Docker, тесты, уверенная защита.
* **Команда**: чёткие задачи и распределение работ.
* **Заказчик-персона**: простота UI и корректность финансовой модели.

---

## 8. Риски, связанные со стейкхолдерами

| Риск                                   | Влияние | Реакция                                  |
| -------------------------------------- | ------- | ---------------------------------------- |
| TL перегружен задачами                 | Высокое | Делегирование, планирование              |
| Отсутствие обратной связи от методиста | Среднее | Частичное самоутверждение документации   |
| UX игнорируется разработчиками         | Среднее | Обязательная интеграция в дизайн-процесс |
| Команда не проводит тестирование       | Высокое | Контроль QA и автоматизация              |

## 9. Итоговая оценка заинтересованных сторон

Наиболее значимые стейкхолдеры:

1. **Team Lead**
2. **Методист / Преподаватель**
3. **Backend и QA/DevOps**
4. **UX/UI Designer**
5. **Эксперт / Комиссия**

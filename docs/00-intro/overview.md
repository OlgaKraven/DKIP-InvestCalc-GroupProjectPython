##  Обзор документации проекта InvestCalc 

Этот документ помогает быстро понять структуру документации проекта **InvestCalc**, а также то, где искать нужную информацию на каждом этапе работы — от постановки задачи до отчётности и презентации.

Документация разделена на логические блоки, которые соответствуют этапам учебного проектирования и разработки информационных систем.

---

##  1. Введение и постановка задачи (`00-intro/`) 

Здесь находится основная вводная информация о проекте:

* **`index.md`** — краткое описание проекта и его цели.
* **`assignment.md`** — формулировка учебного задания.
* **`problem.md`** — описание бизнес-проблемы, которую решает InvestCalc.
* **`domain.md`** — предметная область, контекст и основные понятия.
* **`scenarios.md`** — пользовательские сценарии высокого уровня.
* **`overview.md` (этот файл)** — карта всей документации.

Этот блок помогает студенту понять *зачем* существует проект и *какие задачи* он решает.

---

##  2. Планирование (`01-planning/`) 

Раздел содержит документы, необходимые для предварительной организации работы:

* **`vision.md`** — видение продукта, ключевая ценность.
* **`stakeholders.md`** — заинтересованные стороны.
* **`project-plan.md`** — разбивка проекта на этапы.
* **`gantt.md`** — диаграмма Ганта по проекту.

Этот раздел формирует проектное мышление и учит планированию.

---

##  3. Требования (`02-requirements/`) 

Основной аналитический раздел — фундамент проекта:

* **`requirements-spec.md`** — спецификация требований (ФТ + НФТ).
* **`use-cases.md`** — сценарии использования с диаграммами.
* **`functional.md`** — функциональные требования.
* **`non-functional.md`** — нефункциональные требования (скорость, надёжность, безопасность).
* **`ui-requirements.md`** — требования к интерфейсу.
* **`data-requirements.md`** — требования к данным.
* **`business-rules.md`** — бизнес-правила.
* **`tz.md`** — ТЗ по ГОСТ.

Этот раздел связывает анализ предметной области с будущей архитектурой и реализацией.

---

##  4. Архитектура (`03-architecture/`) 

Раздел описывает структуру системы и ключевые решения:

* **`architecture-overview.md`** — обзор архитектуры приложения.
* **`architecture-decisions.md`** — список архитектурных решений.
* **ADR (`adr-01` … `adr-14`)** — подробные Architecture Decision Records.
* **`c4-context.md`** — контекстная диаграмма системы.
* **`c4-container.md`** — контейнерная диаграмма.
* **`c4-component.md`** — диаграмма компонентов.
* **`sequence-diagrams.md`** — последовательности (например: расчет ROI).
* **`deployment-diagram.md`** — схема развёртывания.

Этот раздел помогает понять, *как* устроена система и почему архитектура выбрана именно такой.

---

## 5. Модель данных (`04-data-model/`)

Содержит полное описание данных проекта:

* **`domain-model.md`** — предметная модель.
* **`er-diagram.md`** — ER-диаграмма сущностей.
* **`db-schema.md`** — логическая структура базы данных.

Этот раздел опирается на требования и используется при разработке API и логики расчётов.

## 6. API (`05-api/`)

Подробное описание интерфейса взаимодействия приложения:

* **`api-overview.md`** — общая структура API.
* **`endpoints-invest.md`** — эндпоинты модуля InvestCalc (TCO, ROI, Payback).
* **`errors-and-status-codes.md`** — коды ошибок.
* **`api-versioning.md`** — политика версионирования API.
* **`openapi-link.md`** — доступ к Swagger, ReDoc и openapi.json.

Этот раздел используется для разработчиков, тестировщиков и интеграторов.

## 12. Как пользоваться документацией

Рекомендуемый порядок для студента:

1. Начать с `00-intro/overview.md`, `problem.md`, `domain.md`.
2. Перейти к требованиям: `requirements-spec.md`, `use-cases.md`.
3. Изучить архитектуру: `architecture-overview.md`, C4-диаграммы.
4. Посмотреть модель данных и API.
5. Перейти к тестированию и DevOps.
6. Перед сдачей отчёта — блок `09-report/`.
